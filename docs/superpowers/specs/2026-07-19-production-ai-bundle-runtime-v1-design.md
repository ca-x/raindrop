## 1. Objective

Turn the already tested `raindrop.ai-content@1.0.0` guest and content worker into a real production path. Every Raindrop binary must embed component bytes, a canonical manifest, its SHA-256 digest, an Ed25519 signature, and the matching public key. Startup must verify and compile that exact bundle, synchronize the installation row after setup becomes ready, compose the HTTPS provider client and official processor when the configured provider keyring is available, and supervise the eight-lane content runtime alongside the existing Feed runtime.

This slice makes the claim-to-artifact path discoverable and runnable from the production process. It does not yet add provider/plugin/content HTTP APIs, enqueue buttons, lifecycle delivery, MCP transport, or the Reader AI sidecar.

## 2. Delegated assumptions

1. The user delegated internal review and explicitly prohibited sub-agent development. The main Agent self-approves this bounded slice after one DDIA/security/contract review.
2. Rust remains edition 2024 with MSRV 1.94.0. `wasm32-unknown-unknown` becomes a declared repository toolchain target because every binary build must produce the embedded guest.
3. A tagged binary or Docker publication is an official release build. It must set `RAINDROP_REQUIRE_OFFICIAL_PLUGIN_SIGNATURE=1` and provide `RAINDROP_OFFICIAL_PLUGIN_SIGNING_SEED` from a GitHub Actions secret.
4. Ordinary local builds and non-release CI use an explicit development trust root derived from a public domain-separated string. It is not secret material and the binary exposes that its embedded signature mode is `DEVELOPMENT`.
5. The official release key ID remains `raindrop-release-2026`. The build consumes a base64url-no-pad 32-byte Ed25519 seed and never writes the seed to `OUT_DIR`, logs, Docker layers, caches, artifacts, or the final binary.
6. Provider secret key configuration remains optional. The embedded plugin installation is synchronized even without a provider keyring, but content lanes remain inert and no temporary/random encryption key is generated.
7. Setup may transition from `FULL` or `ADMIN_ONLY` to `READY` without process restart. The content bootstrap runtime therefore follows the same database-authoritative polling model as the Feed runtime.

## 4. Project structure

```text
build.rs
build/official_ai.rs
  deterministic guest build, componentization, manifest finalization, signing

src/plugins/bundled.rs
  include generated assets, verify embedded public-key/signature/digest, compile component

src/content/worker/production.rs
  wait for setup, sync installation, build provider/plugin/content composition, run lanes

src/background.rs
  supervise Feed runtime and production Content runtime under one shutdown handle

tests/embedded_official_ai_bundle.rs
tests/content_runtime_production.rs
  release asset and setup/startup lifecycle proofs
```

Generated bytes live only under Cargo `OUT_DIR`; no compiled `.wasm`, release private seed, or user-writable component path is committed.

## 5. Build and signing contract

`build.rs` keeps the existing Web bundle gate and delegates official component generation to `build/official_ai.rs`. It emits `rerun-if-changed` for the guest manifest/source, committed WIT/contracts, and manifest template, plus `rerun-if-env-changed` for the three signing variables.

The helper invokes the current Cargo executable with:

```bash
cargo build --locked \
  --manifest-path plugins/official/raindrop-ai-content/Cargo.toml \
  --target wasm32-unknown-unknown \
  --release
```

The nested build uses a private target directory under `OUT_DIR` and removes inherited root target/rustflags that could redirect or contaminate the guest build. The resulting core module is componentized twice with `wit_component::ComponentEncoder`; byte inequality is a build failure. The component must validate and contain no `wasi:` import.

The build computes lower-hex SHA-256 over the component and finalizes the committed manifest template by:

1. replacing `componentDigest.valueSource` with `componentDigest.value`;
2. replacing `signature.valueSource` with an Ed25519 `signature.value`;
3. preserving the exact official identity, ABI, operations, capabilities, schemas, and lifecycle subscription;
4. recursively sorting object keys and serializing compact canonical JSON.

The signature payload remains the frozen v1 frame:

```text
UTF8("raindrop.plugin-signature.v1")
u64be(len(canonicalManifestWithoutSignatureValue))
canonicalManifestWithoutSignatureValue
u64be(len(componentDigestLowerHex))
UTF8(componentDigestLowerHex)
```

Build outputs:

```text
$OUT_DIR/raindrop-ai-content.component.wasm
$OUT_DIR/raindrop-ai-content.manifest.json
$OUT_DIR/raindrop-ai-content.public-key.bin
```

The seed is held only in a zeroized build-process buffer. The public key, signature, digest, manifest, and component are intentionally embedded.

### Signing modes

| Mode | Trigger | Key ID | Publication rule |
| --- | --- | --- | --- |
| `DEVELOPMENT` | no signing seed and official mode not required | `raindrop-development-2026` | allowed for local builds, tests, CI smoke only |
| `OFFICIAL` | valid signing seed supplied | `raindrop-release-2026` | required for tag binary and Docker publication |

If official mode is required and the seed is absent, malformed, non-canonical, or not exactly 32 bytes, the build fails without printing the value. Release binaries and Docker workflows set the requirement. Docker passes the seed through a BuildKit secret mount, never a build argument.

## 6. Embedded bundle boundary

`EmbeddedOfficialAiPlugin::load()` constructs the one trusted key from the embedded public key and runs `BundledOfficialPlugin::verify` over the embedded manifest/component. It additionally requires the manifest key ID to match the build-emitted key ID and exposes only:

- verified `BundledOfficialPlugin` metadata;
- immutable component bytes;
- `EmbeddedSignatureMode::{Development,Official}`;
- `compile(&PluginRuntime) -> Result<CompiledPlugin, PluginRuntimeError>`.

No API accepts a path, environment override, network URL, uploaded bytes, alternate key, or manifest. A valid signature proves the embedded release origin; model/plugin output remains untrusted and still passes the existing WIT/artifact validation.

## 7. Production content bootstrap

`ProductionContentRuntime` owns:

- `SetupService`;
- validated provider-secret entries copied from loaded configuration;
- one verified embedded bundle and compiled component;
- one `ContentRuntimeHandle` control channel.

`new` creates the hardened `PluginRuntime`, verifies and compiles the embedded component before any database installation row can be written. `run` then:

1. waits until `SetupService` is `READY`, polling at one second and observing shutdown;
2. reads the attached `DatabaseConnection`;
3. calls `PluginRegistryRepository::sync_bundled` before starting claims;
4. retries only transient database/revision failures with a bounded one-second wait; structural corruption or identity failures stop supervision with a redacted error;
5. when the provider keyring is empty, logs one metadata-only disabled message and waits for shutdown without claiming jobs;
6. otherwise constructs `ProviderSecretKeyring`, `ProviderRepository`, `HttpsProviderTransport`, `ProviderClient`, `ProviderAiBroker`, `OfficialAiProcessor`, and the existing eight-lane `ContentRuntime`;
7. reuses the outer handle's notify/shutdown control so future enqueue APIs can wake work immediately.

The database remains the record system. Component verification and provider transport construction hold no database transaction. `sync_bundled` is a short idempotent transaction. All Wasm/provider execution stays in the existing claim/heartbeat/fenced-terminal path.

## 8. Background supervision and shutdown

`BackgroundRuntime` supervises exactly two child runtimes:

- Feed runtime;
- production Content runtime.

`BackgroundRuntimeHandle::shutdown()` marks group shutdown first, then requests both child shutdowns. During normal signal/server shutdown the supervisor waits for both children and preserves their typed failures. If either child returns before group shutdown, the supervisor stops and joins the sibling, then returns `FeedStoppedUnexpectedly`, `ContentStoppedUnexpectedly`, or a redacted typed child failure. The HTTP server coordinator continues to stop background work before server drain.

`AppState` receives both production handles. Existing feed notification behavior remains unchanged; the content handle is present for the next enqueue/API slice without adding speculative routes.

## 9. DDIA review

- Record system: `plugin_installations`, content jobs/attempts/artifacts, and provider rows remain authoritative. Embedded bytes are immutable release inputs, not mutable database state.
- Idempotency: startup synchronization converges on one plugin row; component version/digest changes use the existing revision CAS; claims/artifacts retain their existing stable identities.
- Safety before liveness: an invalid embedded bundle never executes or reaches the installation transaction. Transient sync errors retry; corrupt identity fails closed.
- Setup transition: in-memory polling is only a liveness hint. The runtime re-reads readiness and obtains the database from `SetupService` before synchronization.
- Shutdown: new claims stop before the HTTP server drains; abandoned in-flight work recovers through lease expiry and fencing.

## 10. Security threat model

| Boundary | Threat | Control |
| --- | --- | --- |
| signing seed -> build | disclosure through logs, artifacts, layers, cache | GitHub secret or BuildKit secret mount; no command echo; zeroized buffer; never written |
| template/component -> embedded bundle | tampering, wrong ABI, nondeterminism | canonical manifest, SHA-256, Ed25519, deterministic double componentization, runtime verification |
| local build -> published artifact | accidental development signature release | explicit `RAINDROP_REQUIRE_OFFICIAL_PLUGIN_SIGNATURE=1` in both tag workflows; release-contract gate |
| embedded bytes -> runtime | arbitrary local plugin substitution | compile-time `include_bytes!`; no path/upload/network source; exact key ID and signature verification |
| provider configuration -> worker | missing/invalid secret keyring | loader validation; absent keyring keeps content runtime inert; no generated fallback key |
| child runtime -> process | silent task death | typed group supervision; sibling shutdown; server termination on unexpected exit |

No Feed text, prompt, model output, provider credential, signature seed, or component bytes enter error formatting. Logs contain only fixed state labels, version/digest metadata already public in the manifest, and stable error kinds.

## 11. Testing strategy

1. Build contract tests prove development-mode assets verify, compile, match the frozen descriptor, contain no WASI import, and expose the expected digest/key ID without seed material.
2. A forced-official subprocess/build probe proves an absent or malformed seed fails with a redacted error.
3. Production runtime tests prove setup-required shutdown is inert, setup completion synchronizes the installation without restart, an empty provider keyring claims no jobs, and a valid keyring starts/stops lanes.
4. Background supervisor tests prove signal/server shutdown stops both children first and any unexpected child completion stops its sibling and the server.
5. Release-contract tests freeze wasm target installation, official signing requirements, GitHub secret use, and Docker BuildKit secret mounts.
6. Existing component, provider, plugin registry, content worker, full workspace, release embedding, container smoke, and Playwright suites remain green.

## 12. Boundaries

### Always

- Build the official guest through the committed WIT world and no-WASI component path.
- Verify digest/signature/identity before DB synchronization or Wasmtime execution.
- Keep private seed material out of files, logs, caches, layers, artifacts, and source.
- Preserve setup-without-restart, database fencing, and graceful shutdown behavior.

### Internally decide and document

- Exact private module/type names and redacted error kinds.
- Retry logging cadence for transient startup synchronization failures.
- Whether the development signing mode produces a warning or info log, provided it is never silent in a running binary.

### Never

- Commit a release seed or production signature placeholder.
- Accept user-controlled plugin paths, manifests, keys, or component bytes.
- Generate a provider encryption key when configuration is absent.
- Start content lanes before setup readiness and installation synchronization.
- Add provider/plugin/content APIs, lifecycle dispatch, MCP transport, Reader AI UI, or third-party distribution in this slice.
- Use sub-agents or `git add -A`.

## 13. Completion criteria

1. Debug/local and CI binaries contain a verifiable development-signed official component and identify that mode.
2. Tag binary and Docker workflows cannot publish unless the official signing seed is supplied through protected secret channels.
3. Startup verifies/compiles the embedded bundle and idempotently synchronizes `raindrop.ai-content` after both environment-driven and setup-wizard initialization.
4. With a configured provider keyring, the production process runs the existing eight content lanes through the real HTTPS provider/Wasm composition; without it, RSS remains available and content claims remain untouched.
5. Feed and Content runtimes are jointly supervised and shut down before HTTP drain.
6. Release embedding, container smoke, current-stable, Windows, three-database, Rust, Web, and E2E gates pass without lockfile drift or secret leakage.
